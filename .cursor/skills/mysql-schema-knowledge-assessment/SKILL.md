# MySQL 表结构文档化与知识沉淀方案评估

## 目标

在**用户可提供 JDBC 连接**的前提下：

1. 以**只读**方式连接数据库，遍历限定范围内的表结构（列、类型、主键/索引摘要等）。
2. 将结果整理为**结构化文档**（Markdown 或等价），便于人读与后续自动化。
3. 调用大模型对「如何沉淀进知识库」做**方案评估**（非直接在生产库执行写入）。
4. **前提**：表数量上限、支持的库/表范围、是否含敏感数据等限制，必须在**大模型判断是否支持**的输出中**逐条明确写出**（不得省略为笼统「可以」）。

---

## 可行性结论（供 Agent 复述给用户）

| 维度 | 结论 |
|------|------|
| **技术可行性** | **可行**。`information_schema` + 只读账号即可抽取元数据；与本仓库 `SqlQueryService` / `MysqlContextService` 依赖的 schema 摘要思路一致。 |
| **沉淀路径** | 元数据本身可作为**文本知识**走主动学习/文档通道；业务数据行需另循「受控 SQL + 行数上限」与 `qa.assistant.max-structured-ingest-rows` 等策略，与「仅 schema」评估分离说明。 |
| **主要风险** | 连接串与凭据泄漏、误连生产、大表 `COUNT`/全表扫描压力、PII/合规；须网络隔离、只读、限表数、避免自动 DDL/DML。 |
| **不适合** | 无只读权限、无法圈定 schema、表数量无上限承诺、或要求「模型自动改生产库结构」的一键落库——**不应**由本 skill 默认可执行。 |

若用户环境不满足「只读 + 可圈定范围 + 表数上限」，Agent SHALL 判定为**当前方案不支持或需先补前置条件**，并列出缺失项。

---

## 强制执行的前置约束（须在文档与模型提示中写明）

Agent 在生成**给模型的系统/用户说明**时，SHALL 包含以下可核查条款（可照抄下表用语，按用户实际数值填空）：

1. **连接与权限**：仅 **JDBC 只读**；禁止 `INSERT`/`UPDATE`/`DELETE`/`DDL`；本次任务**仅元数据**，不批量导出业务行（除非用户单独授权且另有过滤与行数策略）。
2. **范围圈定**：仅扫描 schema `______`；表名白名单或前缀 `______`（若无白名单，则必须声明「仅前 N 张业务表」的规则及排序依据，例如按字母序）。
3. **表数量硬上限**：本次最多处理 **N** 张表（N 由用户或项目配置给出；若未给出，Agent SHALL 要求用户确认后再继续，默认建议 **≤15** 与对话前提一致）。
4. **行数与规模**：若评估涉及「带数据沉淀」，须声明与 **`qa.assistant.max-structured-ingest-rows`**（或项目约定上限）的关系，并说明超限时的策略（拒绝 / 抽样 / 分页，仅评估不执行）。
5. **敏感与合规**：是否可能含个人信息/商业秘密；若无法排除，模型评估 SHALL 建议「脱敏后再入向量/文档」或「仅保留 schema 不落数据」。

## 工作流程（Agent 执行顺序）

### 1. 收集与校验

- 取得 JDBC URL、用户名、密码（**不得**写入仓库或提交的明文配置文件；优先环境变量或一次性会话）。
- 确认：目标 **schema 名**、**表数量上限 N**、是否**仅结构**、是否允许 **COUNT** 类语句（若不允许，文档中不写行数仅写结构）。

### 2. 只读拉取元数据

- 使用 `information_schema.tables` + `information_schema.columns`（及可选 `statistics`）过滤 `table_schema` 与表名范围。
- 对每张表输出：表注释、列名、类型、可空性、主键/唯一键摘要；**不要**默认 `SELECT *` 拉全表数据。

### 3. 整理文档

输出一篇 **Markdown《数据库结构说明》**，建议章节：

- 连接范围与只读声明  
- 表清单（≤N）与用途一句话（推断须标注「推断」）  
- 每张表字段表  
- 关系与外键（若有）  
- **不建议模型臆造**的业务规则（无证据不写死）

### 4. 调用模型做「沉淀方案评估」

将下列 **固定前缀** 并入用户消息或 system，再附上《数据库结构说明》全文：

```text
你是知识库接入顾问。基于以下只读表结构文档，评估「如何沉淀进知识库」并判断是否支持接入。

你必须在回答最前面用编号列表明确写出（不可省略）：
1) 本次评估假设的 schema 与表范围；
2) 表数量上限 N 及当前文档实际覆盖的表数；
3) 是否仅元数据、是否涉及业务行与行数策略；
4) 判定结论：支持 / 条件支持 / 不支持，并逐条对应到上述前提；
5) 若支持：建议沉淀路径（文档/向量/结构化 SQL 通道/主动学习文本等）及先后顺序与风险；
6) 若不支持或条件支持：列出必须先满足的条件。

禁止建议执行未授权的写操作或绕过只读权限。
```

### 5. 与当前 demo 仓库对齐（可选）

若工作区为 `demo` 问答项目：说明与 `qa.assistant.mysql-schema`、`SqlQueryService`、`MysqlContextService`、`ActiveLearningService`、`qa_pending_knowledge` 等的关系；**不要**假设表名一定为 `company`/`employee`。

**应用内 HTTP（Phase 1 + 2）**：只读目录、可选大模型评估（`assess=true`）、可选持久化（`persist=true`，载荷为 `combinedMarkdown`）见 `openspec/design/mysql-schema-active-learning-pipeline.md`，接口 `POST /qa/mysql/schema-catalog`（仅使用已配置数据源，不接受请求体中的任意 JDBC）。

## 反模式

- 把用户提供的密码写入 `application.properties` 或 git 跟踪文件。  
- 无上限遍历成百上千张表或大表全量采样。  
- 模型输出「可直接执行」的 DDL/DML 而未经过人工与权限审查。  
- 评估结论中**不写明**表数上限与范围圈定（违反本 skill 核心要求）。
