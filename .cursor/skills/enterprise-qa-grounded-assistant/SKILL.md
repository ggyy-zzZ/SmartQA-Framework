# Enterprise QA Grounded Assistant

## 适用场景

- 用户希望把企业知识库问答从 Demo 改造成可上线方案。
- 用户反馈回答不准、幻觉多、文档很大（如 `enterprise.txt`）且质量参差。
- 用户要求“越用越聪明”的沉淀闭环。

## 快速目标

在不强依赖复杂 RAG 的前提下，先实现四件事：

1. 回答可追溯（必须给证据来源）
2. 证据不足可拒答（降低幻觉）
3. 每次问答可回放（日志与反馈）
4. 高质量问答可沉淀（候选知识 -> 审核发布）

## 执行流程（按优先级）

复制并维护此清单：

```md
实施进度:
- [ ] P0 基线诊断（数据规模、覆盖率、脏数据）
- [ ] P1 数据清洗与结构化（文本 -> JSONL）
- [ ] P2 问答链路改造（先筛选后生成）
- [ ] P3 回答协议升级（evidence/confidence/canAnswer）
- [ ] P4 反馈沉淀闭环（feedback -> knowledge_candidate）
- [ ] P5 评测与验收（离线集 + 在线指标）
```

### P0 基线诊断

- 统计总记录数、坏样本占比、敏感字段占比。
- 校验当前上下文截断是否导致低覆盖。
- 输出《现状诊断》：问题清单 + 优先修复项。

### P1 数据清洗与结构化

默认输入：`src/main/resources/enterprise.txt`

处理规则（默认）：

- 删除/隔离测试样本：`测试`、`主体类型:0`、`主体分类:0`、`经营状态:0`
- 枚举归一：例如 `IN_EXISTENCE -> 存续`
- 敏感脱敏：账号、证照号、电话细节
- 空值规范：空字符串统一为 `null`
- 按公司拆分输出：每家公司一行 JSON（JSONL）

建议产物：

- `data/enterprise_clean.jsonl`
- `data/enterprise_rejects.jsonl`（被丢弃样本及原因）

### P2 问答链路改造

禁止“整文件前 N 字符直接喂模型”，改为：

1. 问题意图识别（状态/股权/证照/开户/地址等）
2. 公司实体定位（公司名/简称/ID/别名）
3. 字段级候选检索（TopK）
4. 基于候选生成答案（附证据）

优先使用可解释检索（关键词/规则/BM25），再决定是否补向量检索。

### P3 回答协议升级

接口返回至少包含：

- `answer`: 最终回答
- `evidence`: 证据数组（来源公司 + 字段 + 片段）
- `confidence`: 0-1
- `canAnswer`: true/false
- `route`: 命中链路（faq/rule/retrieval/generate/reject）

当证据不足时，`canAnswer=false`，并给出“缺失信息”提示。

### P4 反馈沉淀闭环

最小闭环：

1. 记录每轮问答（问题/答案/证据/置信度/路由）
2. 收集用户反馈（useful / not useful）
3. 离线生成候选知识（高分问答抽取）
4. 人工审核后发布为标准知识卡片

### P5 评测与验收

构建评测集（100~300条）：

- 问题
- 标准答案
- 证据字段
- 是否应拒答

基础验收门槛（可调整）：

- 首问解决率 >= 60%
- 有证据回答占比 >= 95%
- 误答率 <= 10%
- 应拒答问题的拒答准确率 >= 90%

## 默认交付物

- 分阶段任务清单与里程碑
- 数据清洗规则与字段映射
- 接口改造方案（请求/响应/错误协议）
- 评测方案与指标看板定义
